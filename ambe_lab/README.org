This crate exists **only** as a private development harness for
reverse-engineering / experimenting with AMBE+2 against a local
[[https://github.com/travisgoodspeed/md380tools][md380-emu]] instance.

- =publish = false= is set in =Cargo.toml=.
- The crate is *excluded* from the workspace's =default-members=, so the
  top-level =cargo build= does not build it. Use =cargo build -p
  ambe_lab= explicitly.
- AMBE+2 is patented (DVSI) and the md380-emu codec was extracted from
  proprietary firmware. **Do not publish, package, or distribute
  binaries built from this crate.** Personal/ham research use only.

* What this is

A library + small CLI that drives the file-based =md380-emu= binary
from [[https://github.com/travisgoodspeed/md380tools][md380tools]] as a
subprocess to encode 8 kHz PCM ↔ raw 8-byte AMBE frames.

Initial purpose is to build a **golden test vector corpus**: feed known
PCM into md380-emu, capture the AMBE bytes, store the pair. Those pairs
are then the reference for any future pure-Rust implementation.

* Building ambe_lab

#+begin_src bash
  cargo build -p ambe_lab
  cargo run -p ambe_lab -- --help
#+end_src

* Setting up md380-emu

The =md380-emu= binary is an ARM-Linux executable that runs the
AMBE+2 routines extracted from the MD-380 radio firmware. On x86 it
runs transparently under =qemu-user=.

** Prerequisites (Debian / Ubuntu)

The emulator runs Linux-userland ARM code under =qemu-user=, so we
need the *Linux* ARM cross-compiler — **not** the bare-metal
=arm-none-eabi-gcc=. The top-level =make= in =md380tools= builds
patches against the radio firmware and *does* want the bare-metal
toolchain; we don't need any of that for the emulator path.

#+begin_src bash
  sudo apt install \
      gcc-arm-linux-gnueabi \
      binutils-arm-linux-gnueabi \
      qemu-user
  pip install --user pycryptodome    # needed by md380-fw's AES unwrap step
#+end_src

** Build

Assuming the repo is cloned at =~/src/md380tools= (this is what the
harness defaults to via =MD380_EMU=):

#+begin_src bash
  cd ~/src/md380tools

  # Fetch and unwrap the D002.032 firmware blob the emulator runs
  # against. Uses curl + Python — no compiler involved.
  make original_D02
  # → produces firmware/unwrapped/D002.032.img

  # Build the emulator itself. Uses arm-linux-gnueabi-gcc.
  cd emulator
  make md380-emu
  # → produces ./md380-emu (static ARM binary, runs under qemu-user)
#+end_src

The =cores/d02032-core.img= RAM image is already checked into the
repo, so no extra fetch is needed.

**Do *not* run the top-level =make=** unless you specifically want the
applet/patch build — it fails without =gcc-arm-none-eabi= installed
and has nothing to do with the emulator.

** Point the harness at the binary

#+begin_src bash
  export MD380_EMU=$HOME/src/md380tools/emulator/md380-emu
#+end_src

Or pass =--binary <path>= to each =encode= / =decode= invocation.

* Subcommands

** =tone=

Generate a sine-wave PCM file (s16le, 8 kHz mono). Doesn't need
md380-emu.

#+begin_src bash
  cargo run -p ambe_lab -- tone --frequency 1000 --seconds 3 tone.pcm
#+end_src

** =encode=

Spawn =md380-emu -e=, feed it the PCM file, store the AMBE bytes (no
=.amb= header; just N×8-byte frames concatenated).

#+begin_src bash
  cargo run -p ambe_lab -- encode tone.pcm tone.ambe
#+end_src

⚠ md380-emu's encoder discards the **first 25 frames (~500 ms)** of
its output as "start noise". Pad your PCM input with at least 0.5 s
of silence at the start if you want the full signal encoded.

** =decode=

Spawn =md380-emu -d=, feed it the AMBE frames, write the decoded PCM.

#+begin_src bash
  cargo run -p ambe_lab -- decode tone.ambe roundtrip.pcm
#+end_src

Listen to a PCM dump with:

#+begin_src bash
  aplay -f S16_LE -r 8000 -c 1 roundtrip.pcm
#+end_src

** =capture-corpus=

Generate a directory of golden =(input.pcm, output.ambe)= pairs plus a
=manifest.json= describing them. These vectors are the validation
oracle for any future pure-Rust codec implementation — every encoder /
decoder we write later must reproduce these bytes (decoder) or stay
close to them (encoder).

#+begin_src bash
  cargo run -p ambe_lab -- capture-corpus corpus/
#+end_src

Default stimulus set (13 entries, ~25 s of audio):

- *Silence*: 2 s
- *Sine tones*: 300 / 500 / 1000 / 2000 / 3000 Hz at amplitude 0.5
- *Amplitude sweep*: 1 kHz at amplitude 0.1 and 0.9 (tests amplitude
  quantization independently of frequency)
- *Frequency sweep*: 300 Hz → 3400 Hz over 3 s
- *White noise*: 2 s each at two RNG seeds
- *Impulse*: full-scale single sample inside a 1 s buffer
- *Dual-tone*: 700 + 1200 Hz simultaneously (multi-pitch test)

Each input is pre-padded with 500 ms of silence by default to absorb
md380-emu's encoder warm-up — control via =--warmup-pad-ms=.

** =dump-ambe=

Read an =.ambe= file and print one line per frame showing the raw 8
bytes, the status byte, and the 49 unpacked voice bits.

#+begin_src bash
  cargo run -p ambe_lab -- dump-ambe corpus/silence-2000ms.ambe --count 5
#+end_src

Output looks like:

#+begin_src text
  100 frames total (800 bytes), printing 0..5
  Frame    0: 00 ab cd ef 12 34 56 78  | status=0  bits[0..48]=10101011 11001101 11101111 00010010 00110100 01010110  bit[48]=0
#+end_src

=--start N= skips the first N frames; =--count M= limits how many are
printed.

** =dump-fields=

Unpack each frame's 49 voice bits into the nine named MBE parameter
indices (b0..b8) and classify the frame kind (=voice= / =silence= /
=tone= / =erasure=). Bit allocation follows mbelib's
=ambe3600x2450.c=.

#+begin_src bash
  cargo run -p ambe_lab -- dump-fields corpus/sine-1000hz-amp50-2000ms.ambe --start 25 --count 5
#+end_src

Output:

#+begin_src text
  99 frames total, printing 25..30
    idx  kind       w0 vuv gain  p24 p58  h5  h6  h7  h8
     25  erasure   122  30   22   64  16   1   2   0   0
     26  erasure   122  30   22   64  16   1   2   0   0
     ...
#+end_src

Fields:

- =w0= — fundamental frequency / pitch index (7 bits, codebook =AmbeW0table=).
  Values 120–123 are erasure markers, 124–125 silence, 126–127 tone.
- =vuv= — voicing decision pattern (5 bits, codebook =AmbeVuv=).
- =gain= — gain Δγ index (5 bits, codebook =AmbeDg=). Walks
  monotonically with input amplitude.
- =p24= — PRBA24 spectral magnitude VQ index (9 bits, =AmbePRBA24=).
- =p58= — PRBA58 spectral magnitude VQ index (7 bits, =AmbePRBA58=).
- =h5= / =h6= / =h7= / =h8= — higher-order coefficients (5/4/4/3 bits).

⚠ md380-emu's encoder emits W0 values in the 120–127 range for pure
sine tones — i.e., it routes them through a "tone mode" rather than
the regular voiced/unvoiced path. mbelib reports these as
erasure/silence/tone. The frames still decode to audible audio via
md380-emu itself, so this is a quirk of how the encoder uses the
reserved W0 codepoints, not a bug in our unpacker.

*** =--interpret=

Pass =--interpret= to translate the raw indices into physical values:
pitch frequency in Hz (=f0=), harmonic count =L=, gain delta =Δγ=, and
the per-band V/UV pattern (V = voiced, u = unvoiced).

#+begin_src bash
  cargo run -p ambe_lab -- dump-fields corpus/noise-amp30-seed1-2000ms.ambe --start 30 --count 5 --interpret
#+end_src

#+begin_src text
  idx  kind       w0  f0(Hz)   L    g       Δγ      V/UV
   30  voice     119    65.0  56   30    5.778  uuuuuuuu
   31  voice      80   117.8  31   30    5.778  uuuuuuuu
   32  voice      92    98.0  37   30    5.778  uuuuuuuu
#+end_src

For non-voice frames (silence / tone / erasure, where W0 ≥ 120 and
has no entry in mbelib's tables) the physical columns show =----=.

Physical accessors are =AmbeFields::pitch_hz= /
=AmbeFields::harmonic_count= / =AmbeFields::gain_delta_log_mag= /
=AmbeFields::voicing_pattern=. Note: =gain_delta_log_mag= returns the
quantized **delta** =Δγ=, not an absolute dB value — the codec
recovers absolute gain via =γ = Δγ + 0.5 · γ_prev=, so it needs
inter-frame state to give true dB.

Tone-frame interpretation (=b0 ∈ {126, 127}=) is not implemented:
mbelib itself doesn't decode tone frames — it just returns a sentinel
=3=. Our codec acknowledges the frame kind but cannot extract a tone
frequency without further reverse engineering of md380-emu's tone-mode
encoding.

** =diff-ambe=

Compare two =.ambe= files frame-by-frame, reporting the Hamming
distance over the 49 voice bits per frame plus the positions that
differ.

#+begin_src bash
  cargo run -p ambe_lab -- diff-ambe \
      corpus/sine-1000hz-amp50-2000ms.ambe \
      corpus/sine-2000hz-amp50-2000ms.ambe \
      --start 25                                 # skip warm-up frames
#+end_src

Output:

#+begin_src text
  Frame   25: dist=12  diffs at bits [3,7,9,11,17,21,...]
  Frame   26: dist=11  diffs at bits [...]
  ...
  Summary over 75 compared frames: 0 identical, 75 differing (avg distance 11.6 / 49 bits)
#+end_src

=--min-distance N= filters out frames whose distance is below N
(handy when most frames are identical and you only want to see where
they actually move).

* Audio format reference

- PCM: signed 16-bit *little-endian*, 8 kHz mono. 20 ms frame = 160
  samples = 320 bytes.
- AMBE frame (md380-emu's =.amb= format): **8 bytes per 20 ms frame**.
  - byte[0]: status (0 = good)
  - bytes[1..7]: 48 voice bits, MSB-first
  - byte[7]: the 49th voice bit in its LSB
  - **No FEC** — this is the raw 49-bit voice frame. The DMR wire
    format (DMRD payload) adds a 23-bit Golay code per frame for a
    total of 72 bits.

The on-disk =.ambe= files this CLI writes contain just the frame
bytes — they do *not* carry md380-emu's 4-byte =".amb"= magic. We
strip it on encode and prepend a different 4-byte placeholder on
decode (because the upstream binary has a magic-check bug that
=exit(1)='s on the correct header).

* AMBE+2 reference material

The codec bit allocation and field semantics used by =src/codec/= are
ported directly from **mbelib** (the canonical open-source decoder).
Other writeups have been useful for understanding the surrounding
context. None of this work is original to this crate — it stands on
ten years of public reverse engineering by others.

** Bit allocation (49 voice bits per 20 ms frame)

| Field | Bits | Voice-bit positions                 | Codebook in mbelib |
|-------+------+-------------------------------------+--------------------|
| b0    |    7 | 0, 1, 2, 3, 37, 38, 39              | =AmbeW0table[120]= |
| b1    |    5 | 4, 5, 6, 7, 35                      | =AmbeVuv[32]=      |
| b2    |    5 | 8, 9, 10, 11, 36                    | =AmbeDg[32]=       |
| b3    |    9 | 12, 13, 14, 15, 16, 17, 18, 19, 40  | =AmbePRBA24[512]=  |
| b4    |    7 | 20, 21, 22, 23, 41, 42, 43          | =AmbePRBA58[128]=  |
| b5    |    5 | 24, 25, 26, 27, 44                  | =AmbeHOCb5[32]=    |
| b6    |    4 | 28, 29, 30, 45                      | =AmbeHOCb6[16]=    |
| b7    |    4 | 31, 32, 33, 46                      | =AmbeHOCb7[16]=    |
| b8    |    3 | 34, 47, 48                          | =AmbeHOCb8[8]=     |

Special b0 values: 120–123 = erasure, 124–125 = silence, 126–127 =
single-tone frame.

** AMBE+2 on the air (DMR / NXDN / dPMR / P25 Phase 2)

Each 20 ms voice frame is 49 voice bits + 23 Golay FEC bits = **72
bits at 3600 bps**. FEC = Golay(24,12) + Golay(23,12) with a
data-dependent scrambler and modulation key. Only the 24 most
error-sensitive voice bits are protected; the other 25 ride bare.
